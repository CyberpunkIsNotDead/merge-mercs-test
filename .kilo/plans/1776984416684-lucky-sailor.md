## Summary of What's Missing

### Critical (must have for the test task)

1. **`POST /auth/guest` endpoint** — The test task explicitly requires this endpoint returning `user_id` and `token`. It was removed during development but is a core requirement.
2. **Per-user state tracking** — Without auth, all clients share one global reward record. The test task implies per-player state ("игатель получает награду").
3. **Client user identification** — "На каждый запуск - свой юзер айди" (each launch gets its own user_id). Client doesn't call any auth endpoint.

### Medium priority

4. **Backend route-level integration tests** — No Supertest tests for the Express routes (auth, daily-rewards GET/POST).
5. **Client API layer tests** — No unit tests for `lib/api.lua` HTTP request/response parsing.
6. **Explicit status text in client** — The task says "статус (можем получить / не можем / почему)". Currently only day indicators show state; no dedicated status message area below the reward info.

### Low priority

7. **Client token persistence** — Without auth, there's nothing to persist. If auth is added, need to decide whether to save/load tokens.
8. **Defold client option** — Task mentions Defold or LÖVE; only LÖVE2D implemented (acceptable choice).

## Recommended Plan

### Phase 1: Restore Auth System (Critical)
- Create `backend/src/routes/auth.ts` with `POST /auth/guest` endpoint
- Create `backend/src/services/authService.ts` for JWT generation/validation  
- Create `backend/src/middleware/authMiddleware.ts` for token verification
- Update Prisma schema to add back `User` model (without tokenHash — use simple API key approach)
- Add `userId` back to `DailyReward` model with FK relationship
- Create migration to restore user-aware schema

### Phase 2: Wire Client to Auth (Critical)
- Client calls `/auth/guest` on startup, stores returned `user_id` + `token`
- All API requests include `Authorization: Bearer <token>` header
- Each launch generates a new guest account (per task requirement)

### Phase 3: Add Status Display to Client (Medium)
- Add status text area below reward info showing:
  - "Ready to claim" when cooldown expired
  - "Come back in X minutes" when in cooldown
  - "Series reset available" when gap > 10 min

### Phase 4: Expand Tests (Medium)
- Add Supertest integration tests for backend routes (`backend/tests/routes.test.ts`)
- Test auth endpoint, daily-rewards GET/POST with various states
- Add unit tests for client `lib/api.lua` HTTP parsing logic
